The Red Repair

2026-06-01

Two days ago I wrote about waking up to a clean stack of agent-authored PRs. The queue was the product, the reviewer was the bottleneck, everything green and waiting on me. I felt good about it.

The stack is still sitting there. It is no longer green. It is fully, uniformly red — and the reddest PR in the pile is the one titled ci: repair GitHub Actions workflows.

THE SELF-REFERENTIAL FAILURE

Here is the state, three days on:

#511  revert Responses-API perf change     FAILURE
#515  remove committed MCP secret           FAILURE
#516  make TS scripts runnable from root     FAILURE
#517  ci: repair GitHub Actions workflows    FAILURE
#518  add shared preflight validation        FAILURE
#519  harden S3 backup script                FAILURE

Every check red — lint, typecheck, test, build, e2e, agent-checks. Including on the PR whose entire job is to make the checks pass. The fix for the broken CI is blocked by the broken CI. The snake found its tail.

This is the failure mode the overnight demo never shows you. Not a bad PR. A correct PR that cannot prove it is correct, because the thing that does the proving is the thing it was sent to fix.

WHY GREEN WENT RED

On Saturday those PRs were green. By Sunday they were red, and not one of them had changed. The diffs were frozen. What moved was the ground underneath them.

That is the tell. When a stack of independent PRs all go red on the same day without a single new commit, you are not looking at six bugs. You are looking at one — upstream, shared, in the workflow definitions or a dependency or a runner image that every job inherits. The PRs did not break. The floor did.

Six red PRs that changed nothing are not six problems. They are one problem wearing six masks. Find the shared dependency before you touch a single diff.

THE AGENT CAN'T SEE THIS

An agent looking at #517 sees a task: repair the workflows. It can read the workflow files, spot a plausible misconfiguration, write a plausible fix, and open a clean PR. What it cannot do is notice that its own PR is red for the same reason the thing it is repairing is red. It does not hold the whole board. It holds the diff.

The repair PR fixing the workflows is exactly the kind of thing that looks done and is not, because "done" here is defined by a green check that the change itself cannot produce. The agent completed the task. The task was not the problem.

WHAT THE HUMAN OWES THE QUEUE

Two dispatches ago I said the verb the human owes a queue is triage, not merge. I was right and I was incomplete. Before triage there is a step I skipped: diagnose the floor.

You cannot triage a stack where every item is red for a reason none of the items contain. Reading #515 on its own merits is wasted motion if #515 is red because the runner can't install dependencies. The first question is not "is this PR good." It is "is the redness real, or is it weather." Six PRs failing identically is weather. You fix the sky, not the planes.

red-stack triage order:
  1. are the failures identical across PRs?  -> suspect shared cause
  2. did any diff change when it went red?    -> if no, look upstream
  3. fix the floor on main first              -> a branch can't fix CI
  4. rebase the stack onto green main
  5. THEN read each PR on its own merits

THE FIX HAS TO LAND OFF THE STACK

The structural lesson: a CI repair that has to pass CI to merge is a deadlock by construction. The repair has to land somewhere the broken gate doesn't guard — directly on main, or through a path that doesn't run the failing workflow, with a human carrying the verification the gate can't. You break the loop from outside the loop. Nothing inside it can.

So #517 in its current form is the wrong shape, even if its diff is the right fix. It is asking the broken system to bless its own cure. The cure has to come from above the system, not through it.

THE QUEUE ROTS

The last thing, and the one that actually stings. Three days ago this was a clean stack I felt good about. I did not triage it that day. I wrote a blog post about how clean it was instead.

A queue is not a static artifact. It decays. Green goes red as the world moves under it. A PR you don't review on the day it lands is not the same PR a week later — same diff, different context, staler base, a CI floor that has since fallen out. The cost of a queue is not just the attention to read it. It is the attention to read it before it rots. I have a stack to prove it.
