The Capability Envelope

May 20, 2026

The lazy way to stop a system from doing harm is to turn it off. The honest way is to make it explain itself every time it acts.

Disabling a feature feels like safety, but it is mostly forgetting. The button is hidden, not removed. The next contributor restores it, the next deploy reintroduces it, the next refactor finds the old path and helpfully wires it back up. Off is a state, not a property.

What you actually want is a system that cannot perform a side effect without naming the reason it should be allowed.

OFF IS NOT A POLICY

An autonomous workflow that occasionally sends messages, charges cards, opens tickets, or pushes commits has one dangerous property: it can act when no human is watching. The instinct, when that goes wrong, is to disable the offending path. Comment it out. Rename the file. Move the launchd plist to .disabled-.

That works for a week. It does not survive contact with the next feature request.

A disabled feature is a feature waiting to be re-enabled by accident.

Real safety is structural. The path is not hidden, it is gated. Every call site has to declare what it is doing and under whose authority. Missing declarations get denied and logged, loudly, in a channel a human will see.

THE ENVELOPE

Treat every outbound side effect like a sealed envelope. Before it leaves the system, somebody has to sign it.

sendThing(target, payload, {
  caps: {
    reason: "authenticated_user"
              | "approved_pending"
              | "auto_ack_inbound",
    actor: "user:42" | "agent:ops" | "system:webhook",
  }
})

Three things matter and only three. Who is asking. Why they are allowed. Which pre-declared reason this fits. Anything else is decoration.

The reason is not free text. It is a finite enum the system can audit. Unknown reason, wrong actor, mismatched context — the call refuses and the refusal is itself an event. Silence is never a success state.

LAYERS, NOT WALLS

One gate is a wall, and walls have exactly one failure mode: someone walks around them. Several gates with different shapes are harder to bypass by accident.

layer 1: operating environment
  - the autonomous queue is physically off
  - launchd plist disabled, script renamed
layer 2: application code
  - every side effect requires a cap token
  - missing token = 403 + audit event
layer 3: human approval flow
  - pending action posts a card with the diff
  - human clicks approve, action fires with cap

None of these layers individually is sufficient. The first can be re-enabled. The second can be passed a bad cap if someone is careless. The third can be tired humans clicking through. Together they form a shape that requires three separate kinds of mistake to defeat.

That is the actual goal: not preventing one mistake, but making the failure mode obviously visible before anything reaches the world.

THE AUDIT IS THE PRODUCT

The interesting artifact from a capability envelope is not the action. The action will fire eventually anyway. The interesting artifact is the record: every call site, every reason, every actor, every denial. A timeline of intent.

Six months later, when a question arises — did the system send that message, who authorized it, was the gate in place, did the cap match — the answer should be a query, not an investigation. That is how you trust an autonomous surface.

The cap log is not a debugging aid. It is the only honest description of what the system actually does.

THE OBLIGATION TO REFUSE

The hardest discipline in this design is not building the gate. It is enforcing the refusal.

A function that takes a cap argument and treats missing cap as "ah, probably fine, default it" is worse than no gate at all, because it creates the illusion of one. The contract has to be: no cap, no action. Wrong cap, no action. Anomalous cap pattern, alert.

The system's job is not to be helpful here.
The system's job is to be obstinate
about the difference between
"requested by a human"
and
"happened to run."

Obstinate is the right adjective. A capability envelope that ever rounds up to permissiveness loses its only purpose.

NAMING INTENT IS THE WORK

The unexpected gift of this pattern is that it forces the codebase to learn its own vocabulary. Suddenly every outbound action has to fit into a small list of permitted reasons. The list itself becomes a kind of constitution.

If a new feature does not fit any existing reason, you have a choice: add a reason and defend it, or admit that the feature should not exist. Both outcomes are good. Both are better than a quietly added force=true flag in some helper nobody reads.

The envelope is not just safety scaffolding. It is the contract between the system and the people who have to trust it.

Off is a state. Constrained is a property.

Build for the property.
