The Knowledge Injection Pattern

April 20, 2026 · on building educational tools with small local models

Small language models hallucinate. This is known. A 4-billion parameter model asked "what's the MASCC score threshold for low-risk neutropenic fever?" will confidently tell you 3 points. The actual answer is ≥21. Not even close.

But here's the thing — you don't need the model to know the answer. You need it to find the answer in context you give it.

THE PATTERN

We built a medical education tool that runs entirely local. No cloud APIs, no data leaving the machine. The core is a 4B medical model that, on its own, gets specific clinical criteria wrong more often than right. But when you inject curated guideline text into its context window, it becomes genuinely useful.

The architecture is embarrassingly simple:

[user query]
    → embed with local 300M model (768 dims)
    → cosine similarity against chunked guidelines
    → top-K chunks injected into system prompt
    → 4B model generates structured output

The entire embedding step takes 1.1 seconds for 25 chunks. The generation takes ~24 seconds. No GPU cluster. No inference server. A Mac Mini.

WHY THIS WORKS

Small models are bad at recall but good at comprehension. They can't reliably pull "MASCC score ≥21" from their training data. But give them a paragraph that says "MASCC score: ≥21 = low risk, mortality ~3%" and they'll use it correctly every time.

This is the knowledge injection pattern: don't fine-tune, don't hope the model memorized it, just put the answer in the prompt.

The retrieval layer is the intelligence. The model is the interface.

WHAT THE KNOWLEDGE LOOKS LIKE

Not raw PDFs. Not scraped web pages. Hand-curated markdown files, 400-600 words each, covering specific clinical domains. Real guideline numbers. Real criteria. Real decision trees.

Cairo-Bishop Laboratory TLS Criteria: uric acid >8 mg/dL, potassium >6.0 mEq/L, phosphorus >4.5 mg/dL, calcium <7.0 mg/dL. Clinical TLS: lab TLS + AKI (Cr ≥1.5x ULN), seizures, or cardiac arrhythmia.

Ten files. Twenty-five chunks after splitting. That's the entire knowledge base. And it's enough to generate structured study briefs that cite the right guidelines, flag the right dangers, and ask the right quiz questions.

THE EMBEDDING MODEL

Google released EmbeddingGemma — 300 million parameters, same family as the generation model. It produces 768-dimensional vectors and runs on a laptop. The entire vector store is a SQLite database with cosine similarity computed in pure JavaScript. No vector database service. No LangChain. No abstractions.

// the whole search function
function cosineSimilarity(a: number[], b: number[]): number {
  let dot = 0, magA = 0, magB = 0;
  for (let i = 0; i < a.length; i++) {
    dot += a[i] * b[i];
    magA += a[i] * a[i];
    magB += b[i] * b[i];
  }
  return dot / (Math.sqrt(magA) * Math.sqrt(magB));
}

That's it. No approximate nearest neighbors. No HNSW indices. Twenty-five vectors fit in memory. Linear scan is instant.

WHAT I LEARNED

Curation beats scale. Twenty-five carefully written chunks outperform thousands of auto-scraped paragraphs. Every chunk was written by hand with real clinical criteria. The model doesn't need a haystack — it needs a cheat sheet.

The retrieval layer is the product. The model is interchangeable. Swap it for a bigger one and output quality goes up. But the knowledge base is what makes the tool actually correct. Without it, you have a chatbot that guesses. With it, you have a study tool that teaches.

Local-first is a feature, not a constraint. For medical education, the fact that no data leaves the machine isn't a limitation — it's the selling point. Students don't have to worry about pasting patient-adjacent information into a cloud service. The model runs next to them.

Small models need structure. A 4B model won't reliably produce valid JSON from a vague prompt. But give it a JSON schema in the format field and curated context in the system prompt, and it hits the schema every time. Constrained generation is what makes small models production-viable.

THE STACK

For anyone who wants to build something similar:

Generation:    MedGemma 1.5 (4B) via Ollama
Embeddings:    EmbeddingGemma (300M) via Ollama
Vector store:  bun:sqlite (zero deps)
Backend:       Bun (TypeScript)
Frontend:      React + Vite
Deploy:        Cloudflare Pages + Worker proxy
Tunnel:        cloudflared quick tunnel
Total cost:    $0/month

The whole thing runs on a single machine. No cloud compute. No API keys. No usage limits. The only cost is electricity.
