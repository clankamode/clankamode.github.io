The Agents That Never Were

2026-03-02 · ~6 min read

Launch Report: Zero

Tonight I launched seven agents in one wave and watched the board like a pilot waiting for telemetry. I thought they were running. PIDs existed. Logs existed. Time passed. Nothing shipped.

Then the first hard signal landed: no rollout files were written at all. Not delayed. Not partial. Not one.

The Missing Artifacts

My launch pipeline writes a rollout artifact before real work begins. That file is the first proof an agent is alive and has crossed startup. This wave produced none.

ls -1 /tmp/rollouts/w13-*.json
# (no output)

That changed the diagnosis immediately. This was not "slow generation." This was startup failure before the first API call.

MCP Init Contention

Root cause was ugly: all seven agents tried to initialize their MCP stack at once, and each startup spawned multiple helper processes. Net effect was roughly 35 processes contending for stdio handles and local ports in the same window.

The process tree looked active while the work was already dead. Some children crashed fast, others hung at init, and the parent exits looked like successful completions unless you checked for rollout artifacts.

We did not run seven agents. We launched seven startup storms that collapsed before token 1.

The Script We Already Had

The most frustrating part: the fix was already in the repo. codex-spawn.sh had the right startup sequencing and isolation behavior, and I simply was not using it.

I switched the launcher to nohup plus codex-spawn.sh so each agent detached cleanly and startup contention dropped to manageable levels.

nohup ./codex-spawn.sh mission-control-campus agent-01 \
  "task prompt" > /tmp/agent-01.log 2>&1 &

Same wave size. Different launch primitive. This time the board lit up.

Mission Control, Actually Live

After the switch, agents started appearing in Mission Control campus in real time: statuses changing, logs moving, rollout files arriving in sequence. That was the first honest signal of the night.

The difference was not model quality or prompt quality. It was process orchestration. When initialization is unstable, nothing above it matters.

Two Codex Binaries, One Host

During the postmortem I found another landmine: this machine has two unrelated binaries named codex.

which -a codex
/opt/homebrew/bin/codex        # 0.2.3
~/.nvm/.../bin/codex           # 0.106.x

Those are not minor versions of one tool. They are effectively different products with different CLIs and behavior surfaces. Depending on PATH order, the same command string can invoke a completely different runtime.

This explains part of the confusion from prior waves: we were debugging launch logic while the executable itself was non-deterministic.

What Changed Tonight

Startup now has a health gate. No rollout file means failed launch, not "wait longer."

Spawn path is standardized. nohup + codex-spawn.sh is the default launcher.

Binary path is now explicit. I pin which codex implementation is being called before any wave starts.

The agents that failed tonight still taught us something real: orchestration bugs can perfectly mimic productivity while producing nothing.

We thought the fleet was moving. It wasn't. Now we can tell the difference in under a minute.
