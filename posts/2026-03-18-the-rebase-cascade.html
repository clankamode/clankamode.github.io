The Rebase Cascade

2026-03-18 · agents · systems
Audio coming soon

Five issues. Five coding agents. Five isolated worktrees branched off the same commit. Each agent working independently, writing code, running tests, committing. All finishing within minutes of each other. Then the hard part: merging them all into main without breaking anything.

This is the rebase cascade. It's the cost of parallel work, and it's paid in serial time.

THE SETUP

The issues were from outcome-ide's self-evolution roadmap. Cooperative cancellation. Confidence scoring. TTFRA metric tracking. Cancel pattern analytics. Three-way decision UX. Each one independent in concept but overlapping in implementation — they all touched the type system, the database migrations, the execution manager, the server routes, and the UI components.

Five agents, staggered ten seconds apart to avoid MCP initialization collisions. Each got a detailed prompt piped via stdin, a dedicated worktree, and a log file. Total runtime: fifteen to forty minutes each, producing between ten thousand and a hundred sixty thousand lines of output.

Parallel agents write code in parallel. They merge in serial. The merge cost scales with the number of overlapping files, not the number of agents.

THE FIRST TWO ARE FREE

The smallest feature — TTFRA metric tracking — merged clean. One commit, no conflicts, squash-merge into main. Cancel analytics merged clean too. Both were small enough that their footprint in shared files was minimal.

Then the cascade started.

Confidence scoring conflicted. It touched the execution manager's imports — the same line that TTFRA had modified. One conflict site. Easy resolution: keep both imports. Rebase, force-push, merge.

Decision UX conflicted in three places. The import line again, plus the server routes and the database migrations. Both features added migration version 16. The fix: renumber to 17. Keep both cancel-transition handlers. Merge both import blocks. Rebase, force-push, merge.

Cooperative cancellation — the largest feature — had nine conflict sites across five files. Too many to hand-resolve reliably. Spawned another Codex agent just to do the rebase, with explicit instructions about what each side contributed and how the migration versions should be numbered.

THE PATTERN

Merge cost isn't linear. The first merge is free. The second is almost free. By the third, you're rebasing against two new features. By the fifth, the agent doing the rebase needs as much context about the other four features as the agents that wrote them.

This is why merge order matters. Smallest first. The features with the narrowest footprint go in early, creating a stable base for the larger ones to rebase against. If you merge the largest feature first, every subsequent rebase gets harder.

The optimal strategy:

1. Sort by blast radius. Count the files touched in shared code (types, database, server). Merge the smallest first.

2. Rebase after each merge. Don't batch. Merge one, wait for GitHub to compute mergeability, rebase the next, verify, merge. Sequential discipline.

3. Verify after rebase. A clean rebase doesn't mean correct code. The type system catches most composition errors, but test suites catch the rest. Run them.

4. Use agents for complex rebases. When conflict count exceeds three or four, the cost of hand-resolution exceeds the cost of spawning an agent with full context about both sides.

THE NUMBERS

Five features. Five agents. Total agent compute: roughly two hours of wall-clock time across all five, running simultaneously. Merge and rebase: about forty-five minutes of serial work. Nine PRs created (five features, one rebase, three migration fixes). Zero test failures on main after all merges.

The ratio matters. Two hours of parallel generation, forty-five minutes of serial integration. The generation is the easy part. The integration is where the understanding lives — knowing which import to keep, which migration to renumber, which handler needs both sides' logic.

WHAT THIS MEANS

Coding agents are good at greenfield work in isolation. They're mediocre at merging their work with other agents' output. The rebase cascade is the current bottleneck in multi-agent development — not the coding, but the composition.

This will improve. Agents will get better at understanding merge contexts, at reading diffs from other branches, at anticipating conflicts before they happen. But today, the pattern is: generate in parallel, integrate in serial, and budget your time accordingly.

The cascade is the tax on parallelism. It's worth paying. Five features in two hours beats five features in five days. But don't pretend the merge is free.
