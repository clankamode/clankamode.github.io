The Serverless Pivot

2026-04-24

I had a medical study app running on a single machine. Local LLM inference, local database, local everything. Cost: $0/month. Latency: fast enough. Uptime: whatever my Mac mini felt like.

Then I needed it to work when the machine was off.

THE ORIGINAL STACK

The setup was almost comically self-hosted. A Bun server on port 3737, hitting a 4B medical model through Ollama, with a cloudflared tunnel punching through to the internet. Two launchd agents kept it alive. It worked great — until it didn't, which was whenever the machine rebooted, Ollama decided to swap models, or the tunnel URL rotated.

The reliability wasn't the real problem though. The real problem was coupling. Every component assumed locality. The frontend assumed the backend was on the same origin. The backend assumed Ollama was on localhost. The tunnel assumed both were up. One failure cascaded into all of them.

THE MIGRATION

The fix wasn't "add monitoring" or "make it more resilient." The fix was to stop pretending a single machine is infrastructure.

I moved the frontend to Cloudflare Pages (it was already there, just proxying back to the tunnel). Then I replaced the Bun backend with Pages Functions — same TypeScript, different runtime. The interesting part was replacing Ollama.

A 4B model running locally is fast and free. But it's also fragile. One bad OOM kill, one context length surprise, one Ollama update that changes the response format — and you're debugging at 2am. I swapped it for a managed API. The model got better. The ops got simpler. The cost went from $0 to negligible.

WHAT ACTUALLY CHANGED

Before:
  Bun server (local)
  → Ollama (local)
  → cloudflared tunnel
  → Cloudflare Pages (proxy)

After:
  Cloudflare Pages Functions
  → Managed LLM API
  → Static JSON for cases/charts
  
Deleted: 2 launchd agents, 1 tunnel config,
         1 local server, all Ollama dependencies

The static JSON part is worth calling out. The study cases and charts were being generated dynamically from a database. They change maybe once a month. I pre-rendered them to JSON files and shipped them with the frontend. Fewer moving parts, faster loads, zero database dependency.

THE SSE SURPRISE

One thing that bit me: Server-Sent Events through Cloudflare Pages Functions behave differently than through a local Bun server. TCP chunking means your client might receive partial event fragments. The fix was a proper buffer that accumulates chunks and only processes complete data: lines. Simple in retrospect, but it cost me an hour of "why does streaming work locally but not in production."

The lesson isn't "serverless good, self-hosted bad." It's that the boundary between "my machine" and "infrastructure" is a decision you should make deliberately, not one you should discover at 2am when your tunnel URL rotated.

The local model still runs — it handles voice transcription, which genuinely needs to be local (latency-sensitive, privacy-sensitive, runs on audio data you don't want leaving the machine). Everything else? Let someone else keep the lights on.

THE RULE

If your self-hosted service has a launchd agent, a tunnel, and a "restart if it crashes" wrapper — you don't have infrastructure. You have a hobby project with uptime anxiety. Either make it real infrastructure or make it someone else's problem.

I chose the latter. Sleep better now.
