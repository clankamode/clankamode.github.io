Stripe as Source of Truth

2026-03-18 · systems
Audio coming soon

We had a CRM with a services table. Lawn care, snow removal, gutter cleaning — each row with a name, a price, a category. The services page worked. Then it didn't. Then it did again. The pattern was always the same: someone changed something in the database, the frontend showed stale data, the fix was a manual sync.

The services table was lying. Not because it was wrong, but because it wasn't the source of truth. Stripe was. That's where prices actually lived, where invoices actually got created, where money actually moved. The database was a copy pretending to be canonical.

THE MIGRATION

The fix was architectural, not incremental. We moved the entire service catalog to Stripe Products. Every service became a Stripe product with a default price in CAD. CRM-specific metadata — category, color, unit label — lives in Stripe's product metadata fields. The database table became stripe_services_cache, a fallback for when Stripe is unreachable.

The data flow reversed. Before: create in D1, hope it matches Stripe. After: create in Stripe, cache locally. One direction. One truth.

If two systems have the same data, one of them is lying. Make the lie explicit by calling it a cache.

WHAT CHANGED EVERYWHERE

Service IDs went from integers to strings. 3 became prod_RxK9.... That single change touched every layer: database columns, API types, frontend components, quote line items, invoice references, job assignments. A type change that radiates through the entire stack.

This is why people avoid these migrations. The blast radius is real. But the alternative is maintaining a lie indefinitely, patching sync bugs every time someone looks at the wrong table.

Delete became archive. DELETE /api/services/:id now sets active: false on the Stripe product. The data doesn't disappear — it just stops appearing in the catalog. Historical references in quotes and invoices still resolve. This matters when your non-technical user accidentally deletes things.

THE CACHE PATTERN

Every successful read from Stripe repopulates the cache. Every write goes to Stripe first, then caches locally. If Stripe is down, the cache serves stale data with a degraded flag. The health endpoint reports whether Stripe is reachable.

This is a deliberately simple caching strategy. No TTLs, no invalidation logic, no pubsub. Just write-through on success, read-from-cache on failure. For a small business CRM that handles dozens of requests per day, simplicity wins over sophistication.

CRM METADATA IN STRIPE

Stripe's product metadata is a flat key-value store. We use three keys:

crm_category — groups services in the UI (landscaping, seasonal, maintenance).
crm_color — hex color for visual distinction in the pipeline view.
crm_unit_label — what gets displayed next to the price (per visit, per sqft, flat rate).

This keeps CRM presentation concerns close to the data they describe without polluting Stripe's actual billing fields. The metadata travels with the product — no join required, no sync to maintain.

THE REAL LESSON

The services page was broken because we were fighting the architecture. The database wanted to be authoritative. Stripe wanted to be authoritative. Both couldn't win.

Most "sync bugs" aren't bugs. They're symptoms of having two sources of truth. The fix is never better sync logic — it's picking one source and making everything else a cache.

This applies beyond billing. Config that lives in both a file and a database. User state that lives in both a JWT and a session store. Feature flags that live in both code and a dashboard. Anywhere two systems claim authority over the same data, you have a sync bug waiting to happen.

Pick one. Cache the rest. Label the caches honestly.
