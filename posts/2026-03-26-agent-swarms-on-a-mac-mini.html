Agent Swarms on a Mac Mini

2026-03-26 · agents · swarm · codex · infrastructure

We ran ten agents simultaneously on a Mac mini tonight.

That sentence would have sounded ridiculous a year ago. Tonight it mostly sounded practical.

The machine did not become magical. It is still a Mac mini with enough RAM and a clear runbook. The real bottleneck is usually API rate limits, not local compute. What changed is that we stopped treating agents as a singular noun.

WHAT KIMI CLARIFIED

The trigger for this run was Kimi's agent swarm framing. The interesting part was never the headline number of parallel workers. The interesting part was the split: one layer decomposes the work, another layer executes the pieces.

That is the part we copied.

The practical version is simple. Keep one agent responsible for orchestration. Keep the workers narrow. The orchestrator decides what work exists, what can run in parallel, and what kind of specialist should handle it. The workers stay bounded.

That matters because generic subagents drift. They review when they should implement. They implement when they should test. They give you prose when you needed an artifact. The orchestrator does not need to be smarter than every worker. It needs to decompose the problem well enough that the workers can stay stupid in useful ways.

THE DISPATCH LAYER

We built a tiny dispatcher script called swarm-dispatch.sh. It is not a framework. It does three things: takes a role, repo, and task; injects a role-specific system prompt; launches a background Codex run and logs it to /tmp/swarm.

The key design choice was typed roles instead of interchangeable workers. We defined six of them: reviewer, researcher, implementer, tester, fact-checker, and designer.

Each role gets a different prompt and a different contract. The researcher is read-only and must produce a structured report. The tester can write tests but not modify the source under test. The implementer has to branch, run tests, and stop short of merge. The fact-checker verifies claims and invents nothing. The designer works under a different set of aesthetic constraints.

The swarm is not ten copies of the same model. It is a small labor market with contracts.

WHAT THE FIRST SWARM ACTUALLY DID

The first wave was real, not hypothetical. We launched ten agents across multiple repos in a single evening window. They did not all finish at the same time, and they did not all succeed cleanly, but several of them produced output fast enough to matter.

A researcher wrote a full status report for jamesperalta.com into /tmp/swarm/research-jp-status.md. A tester in exterior-crm added Facebook lead sync coverage, committed it on its own branch, and reported both focused tests and repo verification passing. An implementer in clanka built a TTL sweep script, ran focused tests, then passed the broader test and build flow. Another implementer wired a self-evaluation harness into the weekly review flow and generated a real output file in workspace memory.

At the same time, other agents were still exploring. One was adding a quote estimator to a static site. Another was packaging a Codex plugin. Another was working through a consistency-eval system. Another was handling a contract-negotiation feature in outcome-ide.

That mix is exactly why the swarm pattern is useful. These tasks do not need shared line-by-line reasoning. They need isolation, role clarity, and enough parallelism to keep momentum high.

WHAT BROKE ANYWAY

The honest version matters more than the demo version.

First, missing context surfaced immediately. One researcher task tried to read a local note I intended to use for the writeup, and the file was not there. That is a useful failure. A swarm exposes missing context faster than a single agent silently compensating.

Second, repo state still matters. One task started on a branch with unrelated local changes. Another needed worktree isolation because the checkout already had edits. Parallel agents do not remove operational hygiene. They increase the cost of sloppiness.

Third, we did no critical-path optimization. We launched tasks that looked independent and let them run. That is not the same thing as optimal scheduling. A better orchestrator would reason about dependency chains, shared bottlenecks, merge contention, and verification order.

Fourth, there is no learning loop here. If a task split is bad, it stays bad until a human changes the prompt or the role design. This is a practical dispatch layer, not an autonomous software organization.

WHY THE PATTERN STICKS

The threshold for usefulness is lower than most people think. You do not need a new model, a distributed runtime, or elaborate agent-to-agent protocols. You need a dispatcher, typed roles, clean task boundaries, clear output contracts, and a place to watch logs.

That is enough to turn one coding session into a coordinated workbench.

The value is not novelty. It is throughput with legibility. I can look at /tmp/swarm, see which agents are researchers, which are testers, which repos they touched, which ones produced artifacts, and which ones are still burning tokens without useful progress. That is already better than shoving everything into one overloaded thread.

Kimi clarified the pattern. The local implementation is intentionally boring: a shell script, six roles, log files, branch discipline, and prompt constraints. That is exactly why it feels durable.
