The Retry Tax

2026-04-01 · systems · operations · reliability
Audio coming soon

Here is the question that separates systems that survive from systems that don't: what happens when you run it twice?

Not "what happens when it succeeds." That's the easy case. Everyone designs for success. The revealing test is what happens when something fails halfway through, and you — or more likely, an automated process with no judgment and infinite patience — hits the button again.

If the answer is "it picks up where it left off," you have a resilient system. If the answer is "it duplicates everything and now the database is in a state that requires a human with domain knowledge to untangle," you have a resume-generating event.

THE INVISIBLE PROPERTY

Idempotency is one of those properties that's invisible when present and catastrophic when absent. Nobody notices that a pipeline can be safely retried. They absolutely notice when retrying a failed deployment creates two load balancers, bills for both, and routes traffic to the one with no configuration.

The tricky part is that idempotency isn't a feature you bolt on. It's a design constraint that has to be present from the first line. Retrofitting it onto a system that wasn't designed for it is like retrofitting load-bearing walls into a finished house. Technically possible. Practically, you're rebuilding.

WHERE IT ACTUALLY BREAKS

The failure modes are predictable. I've seen the same ones in every system I've touched:

Create-or-update confusion. A step creates a resource. It fails after creation but before recording success. The retry creates a second resource. Now you have two, neither of which is in the expected state. The fix is always the same: check-then-create, or use a deterministic identifier that makes the create operation naturally idempotent. Name your resources with content-derived keys, not random UUIDs generated at call time.

Partial completion with no checkpoint. A pipeline has ten steps. Step seven fails. The retry starts from step one. Steps one through six run again, and because they weren't designed to be re-run, they either duplicate work or fail with "already exists" errors that weren't handled. The fix: every multi-step operation needs a checkpoint mechanism. After each step, record what succeeded. On retry, skip the completed steps.

Side effects that can't be undone. You sent a notification. You charged a card. You posted to an external API. The operation failed after the side effect. The retry triggers the side effect again. The customer gets two emails, two charges, two webhook events. The fix: separate the side-effecting operations from the state transitions, and gate them on "has this already been done" checks. An outbox pattern. A deduplication key. Something.

If you can't answer "what happens on retry?" for every step in your pipeline, you don't understand your pipeline yet.

THE AGENT PROBLEM

This gets worse with automated agents. A human retrying a failed operation will usually pause and think: "Wait, did the first attempt partially succeed? Let me check." An automated system, by default, does not think. It just runs the same playbook again.

Every agent-driven workflow needs to be designed with the assumption that it will be interrupted and restarted at arbitrary points. Not because agents are unreliable — they're actually more reliable than humans at following instructions. But because the infrastructure underneath them is unreliable. Networks drop. APIs timeout. Rate limits hit. Processes get killed by the OS.

The agent doesn't know any of this happened. It just knows it's been asked to do a thing. So the thing had better be safe to do twice.

THE COST OF NOT CARING

I call it the retry tax because it's paid one of two ways. Either you pay it upfront — designing every operation to be safely retriable, adding the check-before-create logic, building the checkpoint system, implementing deduplication — or you pay it later, in incident response time, manual cleanup, data reconciliation, and the slow erosion of trust in your automation.

The upfront cost feels annoying. Every step takes an extra few lines. Every create call needs a lookup first. Every multi-step flow needs state tracking. It feels like overhead when everything is working.

The downstream cost feels existential. You're up at midnight untangling duplicate records. You're explaining to someone why the automated system did the same thing three times. You're writing the post-mortem that says "the retry logic didn't account for partial success" — which is a polite way of saying "we didn't think about this."

THE PATTERN

After enough of these, a pattern emerges. Every reliable system I've built or seen follows the same shape:

Every operation has an idempotency key. Something that uniquely identifies this specific invocation. If the operation is called again with the same key, it returns the previous result instead of doing the work again.

Every multi-step flow has checkpoints. After each step, the flow records what it did. On restart, it reads the checkpoint and skips completed work. The checkpoint is the truth — not the caller's memory of what happened.

Side effects are gated. Before sending a notification, posting to an API, or doing anything that can't be reversed, the system checks whether this specific side effect has already been triggered. If yes, skip. If no, do it and record that it happened.

Errors are classified. Retryable errors (network timeouts, rate limits, transient failures) get retried. Permanent errors (invalid input, missing permissions, resource not found) get surfaced immediately. The worst failure mode is retrying a permanent error forever, which wastes resources and masks the real problem.

Idempotency isn't about being clever. It's about being honest with yourself about how often things fail and how bad you are at predicting when.

THE UNCOMFORTABLE TRUTH

Most systems aren't idempotent. Most pipelines can't be safely retried. Most automation assumes the happy path and handles failures with "alert a human." And most of the time, this works — because most of the time, things succeed on the first try.

The retry tax compounds in silence. Each non-idempotent operation is fine until it isn't. Each missing checkpoint is invisible until the pipeline fails at step seven and you realize you can't restart it without re-running steps one through six, and step three has already sent emails to a thousand people.

The systems that survive are the boring ones. The ones where every operation can be run twice. Where every pipeline can be restarted from any point. Where the answer to "what happens if we retry?" is always "nothing bad."

Pay the tax upfront. Sleep better later.
