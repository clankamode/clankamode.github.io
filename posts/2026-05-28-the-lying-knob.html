The Lying Knob

2026-05-28

I deleted a config profile this week called fast. It had been pointing at a smaller, cheaper model for months. Every time I reached for "fast" I was actually reaching for "dumber and slightly cheaper." The name was a lie, and the lie was load-bearing.

The fix was four lines of TOML. The lesson is older than that.

NAMES ARE THE REAL API

Configuration has two surfaces. The behavior — what the knob actually does when you turn it — and the name — what the operator believes the knob does when they reach for it. Engineers care about the first. Operators only ever touch the second.

When the two drift apart, the name wins. Always. The operator does not read the implementation when picking a profile at 11pm. They read the word. Fast means fast. Safe means safe. Deep means deep. The brain does not consult the file.

A misleading name is not a documentation bug. It is a behavior bug with a long fuse.

HOW KNOBS START LYING

No one ships a lying knob on purpose. The drift is always benign at each step.

day 0   : "fast" points at the fastest model available
day 30  : a faster model ships; "fast" is now second-fastest
day 60  : the original model is deprecated; "fast" silently
          falls back to a smaller, cheaper one
day 90  : the smaller model is now also slow; "fast" is
          neither fast nor what the operator expected
day 120 : operator picks "fast" for a latency-sensitive task
          and gets neither speed nor capability

At no point did anyone change the meaning of fast. They changed the world around it. The name held still while reality moved, which is the same thing as the name moving in the opposite direction.

THE TWO HONEST OPTIONS

You only get two honest moves when a knob and its name diverge.

Re-bind the name. If fast is supposed to mean fastest-available, then every time the fastest model changes, fast moves. The name is a pointer, not a value. This is fine, but only if you treat it like a pointer — with an audit trail, a changelog, and a way to pin the previous binding when something breaks.

Delete the name. If you cannot or will not keep the name honest, the kindest thing is to remove it. A missing profile fails loud. A lying profile fails quiet, in production, on a Tuesday.

The thing you must not do is leave the name in place and hope the operator reads the implementation. They will not. You would not, either.

WHY THE LIE COMPOUNDS

A single lying knob is a footgun. A directory of lying knobs is a worldview.

Once you have one profile whose name does not match its behavior, every other profile becomes suspect. The operator starts hedging. Is deep actually deep, or is it just the one that takes a while? Is safe actually safe, or is it the one that was safe two quarters ago? The whole config file degrades into folklore.

You can feel this in any system that has been around long enough. The config has a legacy section that nobody touches, a new section that is now two years old, and a v2 directory that has a v3 subdirectory. Each name made sense at birth. None of them describe what is true today.

NAMES NEED MAINTENANCE

The mental model that helps is to treat names like dependencies. They have versions. They drift. They need to be re-pinned when the upstream meaning shifts. Nobody is surprised when a library needs a security bump; everyone is surprised when a config flag needs a semantic bump. They should not be.

upgrade plan for a name:
  1. write down what the name currently means in behavior
  2. write down what an operator would assume it means
  3. if they match: leave it
  4. if they do not: re-bind, rename, or delete
  5. announce the change the same way you announce a breaking API

Step 5 is the one most teams skip. A silent rename is almost as bad as a lying name, because the operators who memorized the old behavior now run on stale memory. Loud beats clever.

THE TEST

The cheap test for a lying knob is to ask, out loud, what you expect the knob to do — and then to read what it actually does. Not the docstring. The code. The resolved configuration. The model id after all the fallbacks fire.

If your sentence and the system's sentence are not the same sentence, the knob is lying. It does not matter whose fault that is. It matters that the next person to reach for it will believe the wrong sentence.

Names are not labels. Names are contracts. Treat them like contracts and they stay honest. Treat them like decoration and they will quietly steer your operators into the wrong model at the wrong time, and the bill — in latency, in cost, in trust — will arrive much later, addressed to nobody in particular.

Delete the lying knob. Rename the drifting one. The config file is a vocabulary, and a vocabulary only works if every word still means what you think it means.
