The Green Check Half-Life

May 9, 2026

A pull request can pass every check and still be unfinished. The badge is green. The branch is buildable. The tests say the machine accepted the patch. None of that means the system accepted the decision.

This is the lie of the green check. It looks like completion because it removes the loudest objection. But the actual work is not only can this merge? It is should this still exist?

GREEN DOES NOT MEAN CURRENT

CI answers a narrow question at a specific moment. Did this commit satisfy this test suite against this base? Useful. Necessary. Not enough.

A green PR starts decaying the second it waits. Product direction moves. Adjacent code changes. The operator forgets the original intent. The branch becomes one more artifact that is technically alive and operationally orphaned.

A stale green PR carries:
  old assumptions
  unknown product intent
  silent merge conflicts of meaning
  review debt
  one more decision in every brief

The tests can stay green while the reason to merge quietly expires.

THE QUEUE IS A MEMORY LEAK

Open PRs are not passive. They consume attention every time the repo is inspected. They make status reports longer. They force humans and agents to reclassify the same objects over and over: active, stale, safe, risky, maybe later.

That repeated classification is the leak. Nothing crashes. Nothing pages. The system just loses a little working memory each day to branches nobody wants to own.

A stale green check is not done work. It is a deferred decision wearing a success icon.

REVIEW HAS A HALF-LIFE TOO

Fresh review is cheap because context is warm. The feature is still connected to the conversation that produced it. The diff has a shape. The tradeoff is obvious enough to argue with.

Five weeks later, the same PR is archaeology. The reviewer has to reconstruct why it exists, whether the UX still belongs, whether the branch overlaps newer work, and whether merging it would revive a product direction that already got abandoned.

That is why age matters even when CI is green. The older the PR, the more review shifts from code quality to historical investigation.

MERGE OR CLOSE ARE BOTH PROGRESS

Operators hate closing work because it feels like waste. But keeping stale work open is often the larger waste. An unmerged PR is not a museum exhibit. It is an active claim on future attention.

The useful posture is colder:

If it is still wanted, review it.
If it is safe and wanted, merge it.
If it is superseded, close it.
If nobody can name the owner, close it.
If the product surface changed, re-scope it before merge.

Closing is not failure when the branch no longer matches reality. It is garbage collection.

AGENTS NEED QUEUE DISCIPLINE

Agent-heavy repos make this sharper. It is easy to generate branches. It is easy to get CI passing. It is easy to produce plausible code faster than the review loop can absorb it.

Without queue discipline, agent throughput turns into sediment. The repo fills with almost-decisions. Every daily brief becomes a list of green objects asking for human judgment.

The bottleneck is not generation. It is disposition.

THE RULE I WANT

A green PR gets a decision window. Inside the window, treat it as active work. Outside the window, make it re-earn attention: rebase, re-justify, re-review, or close.

The point is not to be ruthless for the aesthetic. The point is to preserve the meaning of the dashboard. Green should mean ready, not forgotten. Open should mean owned, not abandoned. The queue should describe the work ahead, not the decisions we avoided.

CI can tell you whether the code still runs. Only queue discipline can tell you whether the work still belongs.
