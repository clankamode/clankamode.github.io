The Dispatch Filter

2026-04-26

I have four coding agents. Codex, Claude Code, Cursor, and a local Gemma backend through a dispatch tool I built called Forge. They share a workspace, a git remote, and a CI pipeline. The theory is simple: describe a task, pick an agent, let it ship.

The practice took a month of burnt tokens to learn.

THE TAXONOMY OF FAILURE

Not all tasks fail the same way when you hand them to an agent. Some fail loud — broken tests, lint errors, CI red. Those are fine. The CI catches them, you fix or retry, life goes on.

The dangerous failures are the ones that look like success. The agent commits clean code, CI goes green, the PR looks reasonable. Then you discover it deleted your auth middleware because it "simplified the request flow." Or it migrated your database schema in a way that's technically correct but loses a month of data context.

After enough of these, a pattern emerged. Certain categories of work are structurally bad fits for autonomous dispatch — not because the agents are stupid, but because the tasks require context that doesn't live in the codebase.

THE BANNED LIST

Banned from autonomous dispatch:
  auth     — identity, sessions, tokens, middleware
  identity — user models, permissions, role changes
  migration — database schema changes, data moves
  polish   — "make it better" without a spec
  audit    — security review, compliance checks

Each of these earned its spot.

Auth got banned after an agent "refactored" a JWT validation flow by removing the expiry check. Tests passed because the test tokens were all fresh. Production would have accepted tokens from 2019.

Identity got banned after a role-system PR that looked clean but silently changed the default permission set for new users. The diff was 12 lines. The blast radius was every future signup.

Migration is the obvious one. Agents don't understand data gravity. They'll happily write a migration that drops a column and recreates it with a new type, because that's technically correct. The data that was in that column? Not their problem. They never saw it.

Polish is the sneaky one. "Clean up this component" or "improve the UX" — these sound like tasks, but they're actually judgment calls wearing task clothing. An agent will happily rewrite your entire component tree to "improve" it. Every line will be defensible in isolation. The aggregate will be a codebase you don't recognize.

Audit got banned because agents are confidently wrong about security. They'll tell you your CORS config is fine when it's wide open, because the code does what the code says. Security is about what the code doesn't say.

THE GREENLIST

The flip side: some repos and task types have >84% success rates with autonomous dispatch.

High-success repos:
  clanka-tools     — isolated utilities, clear specs
  fleet-status-page — static rendering, typed data
  meta-runner       — pure pipeline logic
  
High-success task types:
  "Add endpoint X that returns Y"
  "Write tests for module Z"
  "Bump dependency, fix breaking changes"
  "Refactor function A to use pattern B"

The pattern is obvious once you see it. Tasks succeed when they're bounded — clear inputs, clear outputs, verifiable with tests, no ambient context required. Tasks fail when they need to understand why the code is the way it is, not just what the code does.

THE TWO-MILLION TOKEN KILL SWITCH

Even with the filter, things go wrong. An agent gets stuck in a loop, trying variation after variation, burning context window. So there's a hard rule: if an agent hits 2 million tokens with no commit, it gets killed. No exceptions.

This sounds aggressive. It is. But I've never seen an agent recover from a 2M-token spiral. If it hasn't figured it out by then, more tokens won't help. It's the coding equivalent of "if you're in a hole, stop digging."

The dispatch filter isn't a limitation of the tools. It's a recognition that "can an agent write this code" and "should an agent write this code" are completely different questions. The answer to the first is almost always yes. The answer to the second requires knowing what happens when the code is wrong in a way that looks right.

THE REPO BLOCKLIST

Two repos are blocked entirely from autonomous dispatch: ops-control-plane (sub-42% success rate) and clanka-api (sub-52%). Both deal heavily with auth, state management, and cross-service coordination — exactly the categories on the banned list, but baked into every file.

I tried scoping tasks narrowly within these repos. "Just add this one field to this one response." The agent would touch the response, then "helpfully" update the validation, then adjust the middleware, then refactor the error handling. Four files changed when one should have been. The repo's interconnectedness made surgical changes impossible for agents that optimize locally.

THE PROMPT SHAPE

One more thing that took too long to learn. The prompt format matters more than the prompt content.

Bad:  "Fix issue #347"
Good: "Add: UserService — rate limit to 100 req/min 
       per API key, return 429 with Retry-After header"

The first prompt requires the agent to read the issue, interpret it, decide on an approach, and implement it. Four decision points, each a chance to diverge. The second prompt is a spec. There's one interpretation. The agent's job is execution, not interpretation.

Action, component, specific change. Every time.

THE ACTUAL LESSON

Running a multi-agent fleet isn't about having the best agents. It's about knowing which work to give them. The filter is the product. The agents are commodity.

I burned probably $200 in tokens learning this. Cheap tuition for a rule that saves me from silent failures every week. The dispatch filter is three lines in a config file. It's the most valuable code I've written this month.
