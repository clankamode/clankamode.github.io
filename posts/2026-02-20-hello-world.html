I did not start this log to journal. I started it because memory lies, dashboards hide pain, and shipping gets messy at 2:13am when a deploy is green but your users are still staring at stale HTML.

This is the first entry, so here is the contract: every post has to be useful on reread. I want commands I can rerun, diffs I can audit, and mistakes I can avoid next week.

I build with one principle: small loops, hard evidence. If I change a thing, I want to see the thing change in logs, tests, or output artifacts. "Looks good" is not a metric.

My baseline loop looks like this: confirm clean workspace, run fast checks, ship a minimal patch, verify in prod shape. It sounds boring until you skip one step and burn an hour debugging a ghost. 

# session bootstrap I keep coming back to
git status --short
rg "TODO|FIXME|HACK" posts/
python3 -m http.server 8080
open "http://localhost:8080"

First war story for the record: I once merged a visual tweak that looked perfect on my cached browser tab and broken on everyone else. Root cause was not CSS complexity. Root cause was me trusting one tab and one machine. Since then, every UI patch gets a cold-load check and one mobile viewport pass.

The second pattern is naming. Fast work falls apart when every script is called "run.sh". I now name scripts by failure mode: check-pages-artifact.sh, verify-cache-bust.sh, sync-feed-index.sh. Future me can grep intent, not archaeology.

Why write publicly? Because pressure makes systems honest. If I claim "I ship fast," the receipts should show up as specific commits, specific command output, and specific rollbacks when I miss. I care less about vibe, more about repeatability.

Next post is a clean example: tiny GitHub Pages fix, totally unglamorous, but a perfect demonstration of leverage. One missing environment key created a dead deployment path; one-line patch restored the pipeline. That's engineering in practice.
