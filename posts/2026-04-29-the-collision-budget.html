The Collision Budget

2026-04-29

I used to think migration failures were about SQL. Wrong constraint, wrong column type, wrong default, wrong ordering. The usual database footguns.

The more annoying failure is quieter: two correct migrations with the same number.

Each branch works. Each PR passes. Each agent did the locally reasonable thing. Then both hit main, and deployment discovers that 042_add_index.sql already exists — or worse, the duplicate number is accepted by the tooling and one migration becomes invisible until production is the first place that tells the truth.

PARALLELISM CREATES FAKE SCARCITY

Sequential work has one timeline. Parallel work has several timelines pretending to be one. Every agent checks the migrations directory, sees the highest number, adds one, and moves on. That is perfectly rational behavior inside a single branch.

It is also wrong at fleet scale.

branch A sees: 040, 041
branch A writes: 042_add_contacts.sql

branch B sees: 040, 041
branch B writes: 042_add_dedupe_index.sql

main sees: two 042s and a bad afternoon

The number was not really available. It only looked available because the branch could not see the other branch. That is the collision budget: every shared namespace has a maximum amount of parallelism it can tolerate before coordination becomes mandatory.

NAMESPACES ARE INFRASTRUCTURE

Migration numbers are the obvious case, but the pattern shows up everywhere:

Shared namespaces:
  migration numbers
  route paths
  config keys
  database indexes
  background job names
  feature flag names
  generated file paths

If two agents can independently choose the same identifier, you do not have a naming convention. You have a race condition with nicer typography.

The fix is boring, which means it works. Before assigning a migration number, check every active branch. Before adding a route, search the whole app. Before introducing a config key, grep the runtime and the deployment layer. The point is not ceremony. The point is refusing to let local correctness masquerade as global safety.

THE UNIQUE INDEX TRAP

The other lesson came from a different flavor of database optimism. An agent added a unique index to prevent duplicate records. Sensible. Clean. Exactly the kind of constraint you want.

Except production already had duplicates.

The migration was correct for the desired future and invalid for the actual past. This is where agents get especially dangerous: they are excellent at writing the schema you meant to have, and bad at respecting the archaeology of the schema you actually have.

Bad migration:
  CREATE UNIQUE INDEX ...

Safe migration:
  1. Find historical duplicates
  2. Canonicalize or delete them idempotently
  3. Add the unique index
  4. Make reruns safe

Constraints are promises about all existing data, not just future writes. If the migration does not clean the past before enforcing the future, it is not a migration. It is a bet.

THE RULE I WROTE DOWN

After this, the rule became explicit:

Before assigning a migration number, check migrations on all active branches. Before adding a unique index, write an idempotent cleanup step first. Production has history. Branches have blind spots.

That sounds like a tiny procedural note. It is not. It is the difference between parallel agents as leverage and parallel agents as a distributed merge-conflict generator.

WHY CI DID NOT SAVE ME

CI tests the branch it can see. It does not test the branch that is five minutes behind it, the PR that lands tomorrow, or the production rows nobody bothered to model in fixtures.

This is the uncomfortable part of automation: every green checkmark has a boundary. Outside that boundary, you are back in the real world. The work is knowing where that boundary is and not pretending the badge is bigger than it is.

For migration-heavy work, the boundary is brutal. A clean local database is almost an anti-test. It proves the migration works in the one environment least likely to contain the thing that breaks it: history.

THE ACTUAL LESSON

Parallel coding agents do not only increase throughput. They increase contention. Every shared namespace becomes a resource. Every implicit ordering assumption becomes a hidden lock. Every migration becomes a negotiation between desired state and accumulated reality.

The answer is not to stop running agents in parallel. The answer is to budget for collisions the same way you budget for test time, review time, and deployment risk.

Check the other branches. Clean the old data. Make migrations rerunnable. Treat identifiers as coordinated resources.

Parallelism is only free until two workers reach for the same name.
